BA REQUIREMENTS TEMPLATE - USE CASE ENTRY

Instructions: 
    Fill out this template when gathering requirements for a new use case. 
    Copy the JSON at the bottom, fill in the values, and submit via POST /api/knowledge.

================================================================================
QUICK REFERENCE GUIDE


Essential Fields (Must Fill)
    - type: Always "use_case" for use cases
    - key: Unique identifier (e.g., "uc-user-login")
    - name: Short, descriptive title
    - description: What the system does
    - businessValue: Why this matters to the business
    - primaryActor: Who uses this feature
    - acceptanceCriteria: How we know it's done
    - flows: Step-by-step user interactions

Optional but Recommended
    - technicalSurface: Which code repos/files are affected
    - relationships: Dependencies on other use cases
    - aiMetadata: Complexity and implementation hints
    - tags: For categorization and search

================================================================================
FIELD DESCRIPTIONS


----------------------------------------------------------------------
1. BASIC INFORMATION
----------------------------------------------------------------------

type (Required)
    The entity type - always "use_case"

key (Required)
    Unique identifier for this use case (format: uc-{description})
    Example: "uc-user-login", "uc-create-order"

name (Required)
    Human-readable title (2-5 words, title case)
    Example: "User Login", "Create Purchase Order"

description (Required)
    One clear sentence describing what the system does
    Example: "Allow users to authenticate and access the system securely"

businessValue (Required)
    Why this matters to the business (focus on outcomes, not technical details)
    Example: "Secure access to protected resources and user data"

primaryActor (Required)
    Who initiates or uses this feature
    Example: "End User", "Administrator", "Customer"

----------------------------------------------------------------------
2. STATUS TRACKING
----------------------------------------------------------------------

status.lifecycle (Required)
    Current stage in development
    Options:
        - "idea" - Just an idea, not yet planned
        - "planned" - Approved and scheduled
        - "in_development" - Currently being built
        - "ai_generated" - Code generated by AI, needs review
        - "human_reviewed" - Reviewed and approved by human
        - "completed" - Fully implemented and deployed
    Example: "planned"

status.reviewedByHuman (Required)
    Has a human reviewed this? (true or false)
    Example: false (for new entries)

status.generatedByAI (Required)
    Was this generated by AI? (true or false)
    Example: false (for BA-created entries)

----------------------------------------------------------------------
3. ACCEPTANCE CRITERIA (Required)
----------------------------------------------------------------------

acceptanceCriteria
    List of testable conditions that define "done"
    Example:
        [
          "User can log in with valid email and password",
          "Invalid credentials show clear error message",
          "Successful login redirects to dashboard",
          "Session expires after 30 minutes of inactivity"
        ]
    Tips:
        - Make each criterion testable (can answer yes/no)
        - Focus on user-visible behavior
        - Include both happy path and error cases

----------------------------------------------------------------------
4. USER FLOWS (Required)
----------------------------------------------------------------------

flows
    Step-by-step description of how users interact with the system
    
    Flow Types:
        - "main" - The happy path (normal, successful flow)
        - "alternative" - Valid alternative paths
        - "error" - Error handling and edge cases

    Example:
        [
          {
            "name": "Successful Login",
            "type": "main",
            "steps": [
              "User navigates to /login page",
              "User enters email and password",
              "User clicks 'Login' button",
              "System validates credentials against database",
              "System creates session token",
              "System redirects user to dashboard"
            ]
          },
          {
            "name": "Invalid Credentials",
            "type": "error",
            "steps": [
              "User enters incorrect password",
              "System validates and detects mismatch",
              "System displays error: 'Invalid email or password'",
              "User remains on login page to retry"
            ]
          }
        ]

    Tips:
        - Always include at least one "main" flow
        - Add error flows for common failure scenarios
        - Write steps from the user's perspective
        - Be specific about system actions

----------------------------------------------------------------------
5. TECHNICAL SURFACE (Recommended)
----------------------------------------------------------------------

technicalSurface
    Which parts of the codebase are affected (helps developers and AI know where to make changes)

technicalSurface.frontend

    repos: Which frontend repositories
        Example: ["web-app", "mobile-app"]

    routes: Which URL paths/pages
        Example: ["/login", "/dashboard", "/logout"]

    components: Which UI components
        Example: ["LoginForm", "AuthProvider", "SessionManager"]

technicalSurface.backend

    repos: Which backend repositories
        Example: ["auth-service", "user-service"]

    endpoints: Which API endpoints
        Example: ["/api/auth/login", "/api/auth/logout", "/api/auth/validate"]

    collections: Which database collections/tables are used
        Example:
            [
              {
                "name": "users",
                "purpose": "Store user credentials and profile",
                "operations": ["READ", "UPDATE"]
              },
              {
                "name": "sessions",
                "purpose": "Track active user sessions",
                "operations": ["CREATE", "READ", "DELETE"]
              }
            ]
        Operations: "CREATE", "READ", "UPDATE", "DELETE"

----------------------------------------------------------------------
6. RELATIONSHIPS (Important for Dependencies)
----------------------------------------------------------------------

relationships
    Links to other use cases this depends on or relates to

    Relationship Types:
        - "depends_on" - This use case requires another to exist first
        - "extends" - This builds upon another use case
        - "implements" - This implements a feature
        - "conflicts_with" - This cannot coexist with another
        - "related_to" - Loosely related

    Example:
        [
          {
            "type": "depends_on",
            "targetType": "use_case",
            "targetKey": "uc-user-registration",
            "reason": "Users must be registered before they can log in"
          },
          {
            "type": "related_to",
            "targetType": "use_case",
            "targetKey": "uc-password-reset",
            "reason": "Users may need to reset password if they forget it"
          }
        ]

    Tips:
        - Always specify "depends_on" relationships - this helps determine implementation order
        - Include a clear "reason" to explain why the relationship exists

----------------------------------------------------------------------
7. AI METADATA (Helps AI Implementation)
----------------------------------------------------------------------

aiMetadata.estimatedComplexity
    How difficult is this to implement?
    Options:
        - "low" - Simple CRUD, basic forms, straightforward logic
        - "medium" - Multiple integrations, moderate business logic
        - "high" - Complex algorithms, security-critical, performance-sensitive
    Example: "medium"

aiMetadata.implementationRisk
    What could go wrong during implementation?
    Example:
        [
          "Password hashing must use bcrypt with proper salt",
          "Session tokens must be cryptographically secure",
          "Rate limiting needed to prevent brute force attacks"
        ]

aiMetadata.suggestedOrder
    Priority/sequence number (optional)
    Example: 1 (implement first), 5 (implement fifth)

aiMetadata.testStrategy
    What types of tests are needed?
    Example:
        [
          "Unit test: password validation logic",
          "Integration test: full login flow with database",
          "Security test: SQL injection, XSS prevention",
          "Load test: handle 1000 concurrent logins"
        ]

aiMetadata.nonFunctionalRequirements
    Performance, security, scalability requirements
    Example:
        [
          "Login must complete within 2 seconds",
          "Support 10,000 concurrent users",
          "Comply with GDPR for password storage",
          "99.9% uptime SLA"
        ]

----------------------------------------------------------------------
8. TAGS (For Organization)
----------------------------------------------------------------------

tags
    Keywords for categorization and search (use consistent tags across related use cases)
    Example: ["authentication", "security", "user-management", "mvp"]

HOW TO SUBMIT
================================================================================

Using cURL:
    curl -X POST http://localhost:3000/api/knowledge \
      -H "Content-Type: application/json" \
      -d @your-use-case.json

Using Postman:
    1. Method: POST
    2. URL: http://localhost:3000/api/knowledge
    3. Headers: Content-Type: application/json
    4. Body: Paste your filled JSON

Expected Response:
    {
      "insertedId": "507f1f77bcf86cd799439011",
      "capabilityGenerated": true,
      "capabilityId": "507f1f77bcf86cd799439012"
    }

Success! Your use case is saved, and a capability was auto-generated for AI implementation.

================================================================================
PRO TIPS
================================================================================

1. Start Simple: 
    Fill required fields first, add optional details later

2. Be Specific: 
    Vague descriptions lead to vague implementations

3. Think User-First: 
    Write from the user's perspective, not technical details

4. Dependencies Matter: 
    Always specify "depends_on" relationships - this determines implementation order

5. Test-Driven: 
    Write acceptance criteria as if they're test cases

6. Iterate: 
    You can update the use case later with PUT /api/knowledge
